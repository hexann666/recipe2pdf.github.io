About

The idea for the project came from following question: how would it be possible to keep recipes found online, originally scattered on different platforms and being visualised with different styles, in one place, structure them in the same manner, create own cook book and extend it on the fly every time a new inspiring recipe appears?

The planning phase of the project was structured according to following schedule and contained 6 topics and the topic finding phase.

Planning activities were divided in sprints and progress was documented and tracked in Scrum boards

To sum up the planning phase of this project this website was created, which contains planning documentation (section Blog) and Requirements. Website structure, contents and navigation are described in the blog article Web Technologies

During the planning phase multiple decisions were made, concerning the project setup starting from more wide perspective: scrum setup, strategies for data preparation and analysis, as well as more practical decisions: libraries for data wrangling and tools, that will be used for development and deployment. Some proof of concept were made to ensure, that desired functionality can be implemented with selected tools. Also UX was thought through and a demo of UI was created.

The process flow of the goal application is represented on the flow diagram below from the user perspective.

This flow includes following user-system interactions: web scraping from a URL, provided by user in the UI; using NLP models to classify text and find the chunks, that are important for future cooking process; store them; create a PDF from these text pieces, arranging the elements on the page according to the templates. Second use case, which is independent from the first one in user’s perspective, is generation of recipe recommendations based on typed in ingredients using another ML model. should be created, that will take in the ingredients, turn them into bag of words and compare with stored datasets, gained in the use case 1, choosing the one, that will have highest match of ingredients. In this way data, scraped, cleaned and classified in the UC1 can be used in UC2. More detailed scheme of interactions between UC1, UC2 and system you can find in blog article Data Science methodology and concept, section Modeling.

This project planning allowed to evaluate and simulate all activities, that are needed for an ML project, from an idea to a state, that is ready for actual implementation to start. Variaty of topics, that were analyzed, show, that code is just a small part of such project and much effort is needed to make it accesible to a user. It includes not only UI, but also elements of a workflow, that stay invisible for a user, hence contributing to stable work of the software: development environment, storage space, tools and platforms to deploy the ready software state, tools for project organisation to make development transparant and understandable for the team. It would be a mistake to expect, that only a good code can be enough for a good ML application. As long as it only exists in a Jupyter Notebook, however progressive and useful the output is, it can’t fully achieve it’s goal, as nobody can use it’s results. Providing solid stable infrastructure is what really allows to maximize the benefits and learning and practicing this process is a valuable contribution to my education as Data Scientist.
